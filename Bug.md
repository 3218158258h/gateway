# 网关项目高价值问题复盘（面试版）

> 目标：沉淀“有技术深度、可复用、可讲故事”的问题解决案例。  
> 说明：以下均为项目真实改造点，已删除低价值琐碎修补项。

---

## 1. 多配置源导致运行态不可观测，排障效率低

### 背景（S）
项目同时使用 `gateway.ini / transport.ini / transport_physical.ini / protocols.ini`。  
启动后只能看到零散日志，无法快速确认“最终生效配置”。

### 任务（T）
把启动阶段改造成可审计、可检索、可比对的配置快照体系。

### 行动（A）
- 在 `app_runner` 增加统一快照：
  - `[snapshot] event=startup_config`
  - `[snapshot] event=device_config`
- 在 `app_router` 增加传输配置快照：
  - `[snapshot] event=transport_config`
- 统一日志格式为 `event=... key=value`，便于 grep 和脚本解析。

### 结果（R）
- 配置漂移问题可直接通过快照对比定位。
- 排障从“靠猜配置”转为“看日志验配置”，问题定位链路明显缩短。

### 面试可讲要点
- 不是“多打日志”，而是建立了**运行态配置事实源**。
- 体现了你对可观测性与运维成本的工程意识。

## 2. 设备启动策略“全有或全无”，系统韧性差

### 背景（S）
多设备场景下，只要一个设备初始化失败，进程整体失败退出。

### 任务（T）
改造成“部分成功可运行”，同时保证失败可追踪、行为可控。

### 行动（A）
- `app_runner` 改为“成功集合压缩”：
  - 单设备失败仅记录并跳过
  - 只在 `initialized_device_count == 0` 时整体失败
- `app_router_start` 对启动失败设备做移除，不回滚已成功设备
- 增加汇总日志：
  - `[device] event=bootstrap_summary configured=... initialized=... failed=...`

### 结果（R）
- 系统从“脆性启动”变为“退化可用”。
- 单点设备故障不再拖垮整网关。

### 面试可讲要点
- 这是典型的**边缘系统高可用策略**：允许局部失败、保持主服务在线。
- 体现你能在一致性与可用性之间做工程权衡。

## 3. 协议配置过度兜底，错误被静默吞掉

### 背景（S）
非默认协议配置出错时，旧逻辑会 fallback 默认值，导致“看起来启动成功但行为异常”。

### 任务（T）
建立“默认协议可兜底、非默认协议强校验”的双模式治理。

### 行动（A）
- 在 `app_protocol_config` 增加 `strict_mode` 与结构校验流程：
  - 非默认协议解析异常直接失败
  - 默认协议保留兼容兜底
- 在 `app_device_layer` 细化协议阶段日志：
  - `load_failed / bind_failed / init_command_failed / apply_success`

### 结果（R）
- 配置错误在启动期暴露，不再进入隐性运行时故障。
- 协议问题从“现场不可解释”变成“启动即失败且可定位”。

### 面试可讲要点
- 你做的是**故障前移**，把运行时不确定性转成启动期确定失败。

## 4. 路由回调依赖全局状态，生命周期与并发边界不清晰

### 背景（S）
设备回调通过全局 `g_router` 取上下文，存在停机期悬挂引用风险。

### 任务（T）
去全局耦合，改为显式上下文传递，保证生命周期闭环。

### 行动（A）
- 升级回调签名为 `callback(void *context, void *ptr, int len)`
- 为每个设备创建 `RouterDeviceCallbackContext`
- 在注册/注销/关闭路径显式分配与释放 context
- 删除全局 `g_router`

### 结果（R）
- 回调上下文来源清晰，生命周期可控。
- 降低并发与停机阶段的悬挂指针风险。

### 面试可讲要点
- 典型 C 工程解耦手法：**函数指针 + context**。
- 能讲清楚并发安全和资源所有权。

## 5. 单话题上云无法区分多接口设备数据

### 背景（S）
上行统一话题（MQTT/DDS）下，旧消息只含 `connection_type/id/data`，无法稳定识别来源设备与接口。

### 任务（T）
不拆多话题前提下，实现可区分、可扩展的统一上行格式。

### 行动（A）
- 在 `app_router` 实现统一 Envelope：
  - `schema_version / interface / device_path / protocol_family / protocol_name / timestamp_ms / payload_len / payload`
- `payload` 保留原业务字段，新增元数据用于设备与接口识别
- 设备标识采用 `device_path`，避免逻辑 ID 漂移带来的歧义

### 结果（R）
- 云端可在单话题内完成来源区分与路由分发。
- 为后续多接口扩展和数据治理预留了稳定格式。

### 面试可讲要点
- 这是**协议演进设计**：兼顾兼容性与可扩展性。
- 体现你有“先定消息契约再扩能力”的架构意识。

## 6. I2C/SPI 直接复用串口流式读模型，接口语义错误

### 背景（S）
旧实现统一 `read()` 读线程，套在 I2C/SPI 上会持续报错，且不符合主机事务模型。

### 任务（T）
按接口语义重构采集逻辑，避免“接口抽象伪统一”。

### 行动（A）
- 新增接口专用后台任务：
  - `app_iface_i2c_background_task`
  - `app_iface_spi_background_task`
- I2C/SPI 按配置做周期事务轮询，不再走串口式持续流读
- 非串口路径不再绑定蓝牙串口帧钩子

### 结果（R）
- 接口行为与内核语义对齐，告警噪音显著下降。
- 为 CAN/SPI/I2C 后续专用协议扩展提供了清晰边界。

### 面试可讲要点
- 你识别并修复了“抽象层错位”问题，而非只修表面告警。

## 7. 守护进程异常重启无控制，可能形成重启风暴

### 背景（S）
子进程异常退出后立即重启，崩溃场景会形成高频拉起，放大故障。

### 任务（T）
把守护能力从“能重启”升级为“受控重启”。

### 行动（A）
- 增加崩溃计数窗口与阈值熔断
- 引入指数退避（含上限）
- 启动失败统一处理入口，避免静默失败
- 标准流重定向改为 `dup2 + O_APPEND`，保证日志可靠落盘
- 新增 `daemon.ini` 做守护参数配置化

### 结果（R）
- 避免重启风暴，故障场景更可控。
- 守护策略可按环境调参，无需改代码重编译。

### 面试可讲要点
- 这是典型稳定性治理：**熔断 + 退避 + 配置化**。

## 8. 设备注册表与协议映射存在错位风险

### 背景（S）
`protocols` 列表空项会被普通 split 吞掉，导致后续协议左移错配。

### 任务（T）
支持“按索引占位”的配置语义，明确表达“该设备无私有协议”。

### 行动（A）
- 新增 `parse_aligned_string_list`，保留空槽位
- `device_registry` 下 `protocols` 空项表示“跳过私有协议”
- 设备层增加空协议分支：设备可启动但不应用私有协议

### 结果（R）
- 协议映射不再错位。
- 混合接口场景可平滑配置“有协议/无协议”设备。

### 面试可讲要点
- 你处理的是**配置语义正确性**，而非简单字符串解析问题。

## 9. OTA 升级链路存在多个高风险边界缺陷（已修复）

### 背景（S）
OTA 模块具备完整流程，但关键边界判断不够严谨，存在“误升级、误通过、配置损坏后误信任”等风险。

### 任务（T）
在不重构整体流程的前提下，修复高风险判定缺陷，保证升级决策与失败路径可预期。

### 行动（A）
- 修复版本比较：
  - 从 `strcmp` 改为数字段版本比较，避免 `1.10 < 1.9` 误判。
- 修复签名下载校验：
  - 明确校验签名下载 `CURLcode`，下载失败直接返回，不进入验签流程。
- 强化 boot 配置完整性：
  - 读取时增加 `version + checksum` 校验；
  - 写入时改为 `O_TRUNC` 并 `fsync`，防止脏尾和掉电丢失。
- 修复升级返回码语义：
  - `ota_upgrade` 区分“无更新（1）”与“检查失败（-1）”。
- 增加空固件保护：
  - 固件大小为 0 时安装直接失败，避免除零与非法写分区。

### 结果（R）
- OTA 升级判定从“弱约束”升级为“强边界校验”。
- 失败路径更可解释，日志语义与返回码一致。
- 降低误升级与异常场景下配置损坏风险。

### 面试可讲要点
- 我优先修的是“会导致错误决策”的边界，不是表面流程。
- 这类修复体现的是线上安全性与可恢复性思维。

## 高级架构挑战（预研方向，面试可展开）

> 说明：以下为“下一阶段计划”，用于展示架构能力，不与已落地项混淆。

### A. 下行路由从 connection_type 升级为 device_path + capability

#### 难点
当前下行按 `connection_type` 匹配，遇到同类型多设备时粒度不足，无法稳定定向到具体设备能力。

#### 方案
- 路由键升级为：`device_path + capability + protocol_family`
- 设备注册时上报能力集（如 `read_temp`, `set_relay`, `query_status`）
- 命令 Envelope 增加 `target` 与 `intent` 字段

#### 权衡
- 优点：路由精度高，跨接口一致  
- 成本：协议定义和云端命令格式需要同步升级

#### 落地步骤
1. 定义下行命令 schema v2  
2. 增加 capability registry  
3. 双栈兼容 v1/v2 一段时间后切流

---

### B. 持久化升级为“写入队列 + 批量提交 + 去重键”

#### 难点
高并发写入下，SQLite 单条写事务会成为瓶颈；网络抖动时重试可能造成重复消息。

#### 方案
- 进程内写入队列（内存）
- 周期批量落库（单事务多条提交）
- 增加业务去重键（如 `device_path + timestamp_bucket + hash`）

#### 权衡
- 优点：吞吐提升、IO 抖动更小、重复可控  
- 成本：内存队列与崩溃恢复策略更复杂

#### 落地步骤
1. 增加批写 worker 与 flush watermark  
2. 表结构新增 `dedup_key` 唯一索引  
3. 压测对比单写/批写吞吐与尾延迟

---

### C. 设备层从“每设备线程”演进到 EventLoop/Reactor

#### 难点
每设备线程模型在设备数量增长时会出现线程调度开销与上下文切换放大。

#### 方案
- 串口/CAN 走 `epoll` 事件驱动
- I2C/SPI 走定时事务调度器
- 统一投递到同一消息总线

#### 权衡
- 优点：资源利用率更好，可扩展到更高设备密度  
- 成本：实现复杂度和调试门槛上升

#### 落地步骤
1. 抽象 `iface_poll_ops`  
2. 引入 event loop 线程池  
3. 对比线程模型与 Reactor 模型在 32/64 设备下表现

---

### D. 配置热更新与版本化回滚

#### 难点
当前改配置通常依赖重启，线上变更效率低；错误配置缺少“一键回滚”。

#### 方案
- 配置中心化加载 + 校验 + staged apply
- 失败自动回滚到上一个版本
- 关键配置支持灰度生效（按设备批次）

#### 权衡
- 优点：变更风险降低，线上操作效率提升  
- 成本：需要更严格的配置 schema 与状态机

#### 落地步骤
1. 配置版本号与快照存档  
2. apply 前做 dry-run 校验  
3. 分批应用并监控失败率阈值触发回滚

---

### E. 端到端投递语义：至少一次到“可证明幂等”

#### 难点
网关在离线重发、网络重连、上下游重试时，容易产生重复投递。

#### 方案
- 上行 Envelope 增加 `message_id / sequence / producer_ts`
- 云端与网关同时维护短窗口去重
- 引入 ACK 关联与重放窗口边界

#### 权衡
- 优点：可量化丢重情况，业务侧更容易做一致性保障  
- 成本：协议字段、存储和状态管理复杂度增加

#### 落地步骤
1. 扩展 envelope schema v2  
2. 增加去重缓存与过期策略  
3. 增加“重复率/重放率”观测指标
