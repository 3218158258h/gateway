# Gateway README 2.0（功能正确性优先的测试执行标准）

## 1. 目标与执行原则

本标准用于统一网关测试执行方式，目标是：
- 先完成基础功能完整性与正确性闭环（Phase A）
- 再开展高并发稳定性验证（Phase B）
- 最后做 CPU/功耗与性能优化（Phase C）

执行原则：
1. 先正确性，后性能优化。
2. 每个测试项必须有输入、步骤、可量化成功阈值。
3. 结论仅允许三类：PASS / FAIL / BLOCKED。
4. BLOCKED 仅用于环境阻塞（依赖缺失、资源不可用），不算代码失败。

---

## 2. 统一成功口径（必须先固定）

### 2.1 Phase A（基础功能/通信）默认阈值

说明：本节阈值是实验室基线值，用于回归对比；生产环境应按链路质量和 SLA 重新标定并冻结版本。

| 指标 | 成功阈值 |
| --- | --- |
| 构建成功率 | 100%（`make`、`make -C test` 返回码均为 0） |
| 启动稳定性 | 连续运行 30 分钟无崩溃、无异常退出 |
| 上行正确率（设备->云） | 100%（字段/长度/payload 字节级一致） |
| 下行正确率（云->设备） | 100%（命令可达并可在串口侧观测） |
| 端到端送达率 | >= 99.9%（基础场景建议目标 100%） |
| 重复消息率 | <= 0.1% |
| 乱序率 | 单设备=0；多设备<=0.1% |
| 时延（实验室内网） | P50 < 100ms，P95 < 300ms，P99 < 800ms |
| 持久化一致性 | 数据库记录与发送/重试统计差异=0 |
| 重启恢复 | 网关重启后 60 秒内恢复通信，未确认消息不丢失 |

### 2.2 Phase B（高并发）建议阈值

| 指标 | 成功阈值 |
| --- | --- |
| 长稳稳定性 | 连续 8 小时无崩溃 |
| 丢包率 | <= 0.1% |
| 重复率 | <= 0.2% |
| P99 时延 | <= 1s（目标负载点） |
| 队列行为 | 队列可回落，不持续增长 |
| 故障恢复 | 网络恢复后 5 分钟内回稳 |

### 2.3 Phase C（优化验收）双约束

同等负载下，必须同时满足：
1. 功能回归 100% 通过；
2. 至少一项显著改善：
   - CPU 平均下降 >= 15%，或
   - 功耗下降 >= 10%，或
   - P99 时延下降 >= 20%
3. 其余关键指标（送达率、稳定性、重试率）不劣化超过 5%。

## 3. 依赖、入口与环境闸门（Gate）

### 3.1 依赖清单
- gcc / make
- Python 3
- sqlite3
- MQTT C SDK（`MQTTClient.h` + 对应库）
- Cyclone DDS（`dds/dds.h` + 对应库）
- socat（虚拟串口联调）

### 3.2 仓库标准入口
- 主工程构建：`make`
- DDS 测试构建：`make -C test`
- 网关启动：`./build/bin/gateway app`
- 虚拟串口创建：`./create_virtual_nodes.sh`
- 下位机模拟：`python3 simulate_lower_device.py`
- 串口监控：`./monitor_virtual_port.sh`
- 数据库查看：`python3 read_gateway_db.py`（打印数据库表、消息总数与最新消息记录）

### 3.3 Gate 判定（先过闸门再跑用例）

同时满足以下条件才允许进入 Phase A：
1. 依赖齐全（MQTT/DDS/socat）。
2. 配置与构建一致：`transport.type=dds` 时必须 `make USE_DDS=1 ...`。
3. `serial_devices` 指向真实存在的串口节点（真实或虚拟）。

**Gate 失败结论：BLOCKED。**

## 4. Phase A：基础功能完整性与正确性（执行版）

### A-00 环境闸门检查
- 目的：确认环境可支持端到端验证。
- 命令：
  - `make`
  - `make -C test`
  - `./create_virtual_nodes.sh 1 /tmp/gateway-vdev`
- PASS：
  - `make` 返回码=0；
  - `make -C test` 返回码=0；
  - `./create_virtual_nodes.sh 1 /tmp/gateway-vdev` 返回码=0 且 `/tmp/gateway-vdev/gw0`、`/tmp/gateway-vdev/sim0` 被创建。
- BLOCKED：缺失头文件/库或 `socat` 不存在。

### A-01 构建基线
- 目的：验证主程序与测试程序可构建。
- 步骤：
  1. 执行 `make`
  2. 执行 `make -C test`
- PASS：返回码均为 0。
- FAIL：代码编译错误。
- BLOCKED：外部依赖缺失（如 `MQTTClient.h`、`dds/dds.h`）。

### A-02 启动与配置一致性
- 目的：验证配置加载和传输层初始化路径正确。
- 前置：根据 `gateway.ini` 设置 transport。
- 步骤：
  1. `transport.type=mqtt` 时使用普通构建产物启动。
  2. `transport.type=dds` 时使用 `make USE_DDS=1 DDS_HOME=...` 构建后启动。
  3. 运行 `./build/bin/gateway app`，观察日志 30 分钟。
- PASS：无 ERROR/FATAL 初始化失败，进程持续存活。
- FAIL：初始化失败、崩溃或异常退出。

### A-03 串口链路正确性（无云）
- 目的：验证串口收帧与解析正确。
- 步骤：
  1. `./create_virtual_nodes.sh 3 /tmp/gateway-vdev`
  2. 设置 `[device].serial_devices=/tmp/gateway-vdev/gw0,/tmp/gateway-vdev/gw1,/tmp/gateway-vdev/gw2`
  3. 启动网关。
  4. 向 `sim0` 发送固定帧（`sim0` 与 `gw0` 为同一对虚拟串口）：
     ```bash
     python3 simulate_lower_device.py \
       --port /tmp/gateway-vdev/sim0 \
       --device-id 0001 \
       --payload 01020304 \
       --count 100 \
       --interval 0.2
     ```
  5. 监控串口：`./monitor_virtual_port.sh /tmp/gateway-vdev/gw0`
- PASS：
  - 有效帧识别率 100%；
  - 错帧/非法帧不导致崩溃；
  - 后续帧仍可继续处理。

### A-04 MQTT 基础闭环
- 目的：验证 MQTT 上下行端到端。
- 前置：`transport.type=mqtt`，`[mqtt]` 参数完整。
- 步骤：
  1. 设备侧持续上报（参考 A-03）。
  2. 验证数据发布到 `publish_topic`。
  3. 向 `subscribe_topic` 发送下行命令，观察网关转发到串口。
- PASS：
  - 上下行闭环均成功；
  - 字段/topic/payload 完全匹配；
  - 1000 条样本：送达率 >= 99.9%，重复率 <= 0.1%。

### A-05 DDS 基础闭环
- 目的：验证 DDS 上下行端到端。
- 前置：`transport.type=dds` 且 `make USE_DDS=1 DDS_HOME=...`。
- 步骤：
  1. 构建并运行 `test/` 下 publisher/subscriber。
  2. 核对 topic/type：`GatewayData` / `GatewayCommand`。
  3. 验证网关上下行路径。
- PASS：
  - DDS 上下行各至少一次稳定闭环；
  - 1000 条样本送达率 >= 99.9%；
  - 无域冲突、无类型不匹配。

### A-06 持久化与异常恢复
- 目的：验证重试、落库与恢复一致性。
- 步骤：
  1. 运行网关并产生消息。
  2. 人为制造短断网并恢复。
  3. 重启网关后执行：
     - `python3 read_gateway_db.py --limit 50`
  4. 核对 `messages` 表中 `status/retry_count/create_time/update_time` 演进是否一致。
- PASS：
  - 重试次数不超过配置上限；
  - 无“已发未落库/已落库未发出”差异；
  - 重启后 60 秒内恢复，最终一致性差异=0（定义为 `|发送成功总数 - 数据库中成功状态记录总数|=0`）。

### A-07 多设备基础完整性（非压测）
- 目的：验证多设备并行隔离性。
- 步骤：
  1. 按 1/4/8 路设备分层执行。
  2. 每路发送独立 device-id/payload。
  3. 检查路由方向、身份映射与串口输出。
- PASS：
  - 无串话、无错路由；
  - 8 路场景下仍满足送达率 >= 99.9%，且无崩溃。

## 5. Phase B：高并发与稳定性（详细执行）

### 5.1 压测模型（至少 4 类）
1. **Ramp Up**：逐级加压（找吞吐拐点与时延拐点）。
2. **Sustain**：固定高负载 30~120 分钟（看稳定性与队列趋势）。
3. **Spike**：瞬时 2~5 倍流量（看削峰能力与恢复速度）。
4. **Soak**：8~24 小时长稳（看内存泄漏、抖动、衰退）。

### 5.2 并发维度（建议矩阵）
- 设备数：8 / 16 / 32
- 单设备速率：1 / 5 / 10 msg/s
- 报文大小：小包（16B）/中包（128B）/大包（512B）
- 网络扰动：正常 / 丢包 / 短断 / 抖动

### 5.3 必采集指标
- 吞吐：msg/s（入口/出口）
- 质量：丢包率、重复率、乱序率
- 时延：P50/P95/P99
- 弹性：重连次数、重试率、恢复时间
- 队列：峰值深度、清空时间、是否持续增长
- 存储：数据库写入耗时、失败率
- 稳定性：崩溃/卡死次数（必须为 0）

### 5.4 通过判定
- 连续 8 小时无崩溃；
- 丢包率 <= 0.1%；
- 重复率 <= 0.2%；
- P99 <= 1s（目标负载）；
- 数据库写入耗时 P99 <= 50ms（实验室基线）；
- 队列可回落；
- 故障恢复后 5 分钟内回稳。

## 6. Phase C：CPU/功耗评估与优化闭环

### 6.1 固定场景
- Idle（空载）
- Nominal（目标负载）
- Peak（高压/突发）

### 6.2 指标采集
- CPU：平均、峰值、线程热点
- 内存：RSS、增长斜率（泄漏趋势）
- I/O：磁盘写入速率、等待/fsync 行为
- 网络：收发速率、重传
- 功耗：整机功率（外部功率计）或板级采样

### 6.3 优化顺序（每次只改一个变量）
1. 先定位瓶颈（线程池、序列化、队列、数据库、网络重试）。
2. 单点优化并记录变更前后基线。
3. 回跑 Phase A 核心回归。
4. 回跑 Phase B 关键场景对比。
5. 不达标则回滚或继续迭代。

### 6.4 优化验收
满足第 2.3 节双约束即视为优化成功，否则不合入。
